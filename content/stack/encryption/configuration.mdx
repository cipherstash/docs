# Configuration

The SDK supports two configuration methods: environment variables and programmatic config. Environment variables take precedence over programmatic config.

## Environment variables

The simplest approach. Set these in a `.env` file or your hosting platform:

| Variable | Description | Required |
|---|---|---|
| `CS_WORKSPACE_CRN` | The workspace identifier (CRN format) | Yes |
| `CS_CLIENT_ID` | The client identifier | Yes |
| `CS_CLIENT_KEY` | Client key material used with ZeroKMS | Yes |
| `CS_CLIENT_ACCESS_KEY` | API key for authenticating with the CipherStash API | Yes |
| `CS_CONFIG_PATH` | Temporary path for client configuration | No (default: `/home/{username}/.cipherstash`) |
| `STASH_STACK_LOG` | Log level (`debug`, `info`, `error`) | No (default: `error`) |

Sign up at [cipherstash.com/signup](https://cipherstash.com/signup) and follow the onboarding to generate credentials.

## Programmatic config

Pass config directly when initializing the client. Useful when loading credentials from a secret manager:

```typescript title="protect/index.ts"
import { Encryption, type EncryptionClientConfig } from "@cipherstash/stack"
import { users } from "./schema"

const config: EncryptionClientConfig = {
  schemas: [users],
  config: {
    workspaceCrn: "your-workspace-crn",
    accessKey: "your-access-key",
    clientId: "your-client-id",
    clientKey: "your-client-key",
  },
}

const client = await Encryption(config)
```

## Keysets

[Key Sets](/stack/kms/keysets) are a ZeroKMS primitive for cryptographic isolation.
Each Key Set maintains its own set of data encryption keys â€” data encrypted under one Key Set cannot be decrypted with another.
This is the same primitive that powers [environment isolation in Secrets](/stack/secrets/concepts#environments).

You can specify a Key Set by ID or by name:

### By ID

```typescript title="protect/index.ts"
const client = await Encryption({
  schemas: [users],
  config: {
    workspaceCrn: "your-workspace-crn",
    accessKey: "your-access-key",
    clientId: "your-client-id",
    clientKey: "your-client-key",
    keyset: { id: "123e4567-e89b-12d3-a456-426614174000" },
  },
})
```

### By name

```typescript title="protect/index.ts"
const client = await Encryption({
  schemas: [users],
  config: {
    workspaceCrn: "your-workspace-crn",
    accessKey: "your-access-key",
    clientId: "your-client-id",
    clientKey: "your-client-key",
    keyset: { name: "Company A" },
  },
})
```

For full details on creating and managing Key Sets, see the [Key Sets reference](/stack/kms/keysets).

## Logging

Control log verbosity with the `STASH_STACK_LOG` environment variable:

```bash
STASH_STACK_LOG=error  # debug | info | error (default: error)
```

| Value   | What is logged          |
| ------- | ----------------------- |
| `error` | Errors only (default)   |
| `info`  | Info and errors         |
| `debug` | Debug, info, and errors |

The SDK never logs plaintext data.

## Deploying to production

### File system write permissions

In most hosting environments, set `CS_CONFIG_PATH` to a writable directory:

```bash
CS_CONFIG_PATH=/tmp/.cipherstash
```

This has been tested on [Vercel](https://vercel.com/) and [AWS Lambda](https://aws.amazon.com/lambda/).

### Bundler configuration

`@cipherstash/stack` includes a native FFI module that must be excluded from bundling:

- **Next.js**: See [Bundling â€” Next.js](/stack/encryption/bundling#nextjs)
- **SST / serverless**: See [SST setup](/stack/encryption/sst)
- **Webpack / esbuild**: Add `@cipherstash/stack` to your externals configuration

### Platform requirements

- **Node.js** >= 18
- [Bun](https://bun.sh/) is not currently supported due to incomplete Node-API compatibility
- See [Troubleshooting](/stack/encryption/troubleshooting) for npm lockfile issues on Linux
