# Concepts

CipherStash Secrets is organized around four core resources: workspaces, environments, clients, and API keys. Understanding how they relate helps you design a secure secrets management strategy.

## Workspaces

A **workspace** is the top-level organizational unit. Each workspace gets its own isolated vault for storing encrypted secrets.

- Separate resources by project or team.
- Each workspace has a unique Cloud Resource Name (CRN) and region (e.g., `us-east-1`, `eu-west-1`).
- Secrets stored in one workspace are completely isolated from other workspaces.

## Environments

**Environments** are an SDK-level abstraction over [Keysets](/stack/kms/keysets), the core cryptographic isolation primitive in CipherStash Cloud. Each environment maps directly to its own keyset in ZeroKMS, so secrets in one environment cannot be decrypted with keys from another.

In the CipherStash Dashboard, environments appear as keysets. The `environment` parameter in the SDK resolves to a keyset by name. This is the same keyset primitive used for [tenant isolation](/stack/encryption/configuration#keysets) in the Encryption SDK.

Typical setups include:

| Environment | Purpose |
|---|---|
| `production` | Live service secrets |
| `staging` | Pre-production testing |
| `development` | Local development |

When you initialize the SDK, the `environment` parameter determines which keyset is used.

For local development with device-based auth:

```typescript filename="secrets.ts"
import { Secrets } from "@cipherstash/stack/secrets"

const secrets = new Secrets({
  environment: "development",
})
```

For production with environment variables:

```typescript filename="secrets.ts"
import { Secrets } from "@cipherstash/stack/secrets"

const secrets = new Secrets({
  workspaceCRN: process.env.CS_WORKSPACE_CRN!,
  clientId: process.env.CS_CLIENT_ID!,
  clientKey: process.env.CS_CLIENT_KEY!,
  accessKey: process.env.CS_CLIENT_ACCESS_KEY!,
  environment: "production",
})
```

## Clients

**Clients** represent the services that need access to secrets. Each client receives a unique client key used for authentication with ZeroKMS.

There are two types of clients:

- **Device-backed clients** are created automatically by `npx @cipherstash/cli init` and tied to a developer's identity. Used for local development — no environment variables needed.
- **Machine clients** are created manually in the Dashboard for production and CI/CD. Identified by the `CS_CLIENT_ID` and `CS_CLIENT_KEY` environment variables.

A client is associated with one or more environments. Client keys are shown only once at creation — store them securely. See [Clients](/stack/kms/clients) for more details.

## API keys

**API keys** (also called access keys) authenticate requests to the CipherStash API. They are separate from client keys and control API-level access.

- Set the `CS_CLIENT_ACCESS_KEY` environment variable with your API key.
- Roles control permission scope: `admin` for full access, `member` for limited access.
- Like client keys, API keys are shown only once at creation.

## Credential summary

The following environment variables are required for **production and CI/CD** only.
For local development, device-based auth handles credentials automatically — see [Getting started](/stack/guides/getting-started).

| Variable | Description |
|---|---|
| `CS_WORKSPACE_CRN` | Workspace identifier (CRN format) |
| `CS_CLIENT_ID` | Client identifier |
| `CS_CLIENT_KEY` | Client key material used with ZeroKMS |
| `CS_CLIENT_ACCESS_KEY` | API key for authenticating with the CipherStash API |

See [Going to production](/stack/guides/going-to-production) for a step-by-step guide to setting up production credentials.

## Security model

- **End-to-end encryption** — Values are encrypted locally by the SDK before transmission. CipherStash never sees plaintext.
- **Workspace isolation** — Each workspace has its own vault. No cross-workspace access is possible.
- **Environment isolation** — Each environment maps to a separate [keyset](/stack/kms/keysets) in CipherStash Cloud. A secret encrypted in staging cannot be decrypted with production keys.
- **Zero-trust** — Every request is authenticated and authorized. There is no implicit trust between services.
