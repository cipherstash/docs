Retrieval-augmented generation (RAG) pipelines commonly store sensitive documents alongside vector embeddings. Without encryption, that data is exposed at rest and during retrieval, creating a significant attack surface. CipherStash lets you encrypt sensitive content while preserving the ability to search and retrieve it.

## The problem

RAG architectures typically store:

- **Document chunks:** the original text, often containing PII, financial data, or confidential business information.
- **Metadata:** source references, user associations, access tags.
- **Vector embeddings:** numeric representations used for similarity search.

If any of this is exfiltrated from the database, the plaintext content is immediately exposed. Encryption at rest does not help, because the data is decrypted as soon as it is queried.

<Callout type="info">
CipherStash encrypts the document chunks and metadata, not the embeddings themselves. An embedding is a lossy projection of its source text, and similarity search has to compare embeddings in the clear. Treat embeddings as sensitive: they can leak information about the text they were derived from.
</Callout>

## Encrypting RAG context data

Use the [Stack SDK](/reference/stack) to encrypt sensitive fields before storing them alongside your embeddings.

### Define a schema for your documents

```typescript filename="schema.ts"
import { encryptedTable, encryptedColumn } from "@cipherstash/stack/schema"

export const documents = encryptedTable("documents", {
  content: encryptedColumn("content").freeTextSearch(),
  source: encryptedColumn("source").equality(),
  userId: encryptedColumn("user_id").equality(),
})
```

Each index method declares a capability. `freeTextSearch()` makes the content matchable by token; `equality()` makes a column filterable by exact match. See [searchable encryption](/concepts/searchable-encryption) for what each capability reveals.

### Encrypt before storage

```typescript filename="ingest.ts"
import { Encryption } from "@cipherstash/stack"
import { documents } from "./schema"

const client = await Encryption({ schemas: [documents] })

async function ingestDocument(doc: {
  content: string
  source: string
  userId: string
  embedding: number[]
}) {
  const encrypted = await client.encryptModel(
    { content: doc.content, source: doc.source, userId: doc.userId },
    documents,
  )

  if (encrypted.failure) {
    throw new Error(`Encryption failed: ${encrypted.failure.message}`)
  }

  await db.query(
    `INSERT INTO documents (content, source, user_id, embedding)
     VALUES ($1::public.text_match, $2::public.text_eq, $3::public.text_eq, $4)`,
    [
      encrypted.data.content,
      encrypted.data.source,
      encrypted.data.userId,
      JSON.stringify(doc.embedding),
    ],
  )
}
```

### Decrypt retrieved context

After vector similarity search retrieves the relevant documents, decrypt the content before passing it to the LLM:

```typescript filename="retrieve.ts"
async function retrieveContext(queryEmbedding: number[], topK = 5) {
  const results = await db.query(
    `SELECT content, source FROM documents
     ORDER BY embedding <-> $1
     LIMIT $2`,
    [JSON.stringify(queryEmbedding), topK],
  )

  const decrypted = await client.bulkDecryptModels(results.rows)

  if (decrypted.failure) {
    throw new Error(`Decryption failed: ${decrypted.failure.message}`)
  }

  return decrypted.data
}
```

`bulkDecryptModels` sends every ciphertext in one round trip to ZeroKMS, rather than one call per row. On a `topK` of 5 that hardly matters; on a reranking pass over a few hundred candidates it is the difference between one request and hundreds.

## Searchable encrypted retrieval

When you need to filter documents by metadata before or alongside vector search, combine an encrypted equality predicate with the similarity ordering:

```sql
-- Find documents for a specific user, then rank by similarity
SELECT content, source, embedding
FROM documents
WHERE user_id = $1::public.text_eq
ORDER BY embedding <-> $2
LIMIT 10;
```

The operand must be typed (`$1::public.text_eq`) for the encrypted `=` operator to resolve. See [typed operands](/reference/eql/core-concepts) and [filtering](/reference/eql/filtering).

This filters on encrypted metadata inside PostgreSQL, without decrypting it. Add a functional index over the term extractor so the filter does not become a sequential scan. See [indexes](/reference/eql/indexes).

## Benefits for AI pipelines

- **Sensitive context stays encrypted.** Document chunks containing PII or confidential data are never stored in plaintext.
- **Selective decryption.** Decrypt only what the LLM needs, reducing the exposure surface.
- **Audit trail.** Track who retrieved which documents and when, using [identity-aware encryption](/solutions/provable-access).
